Summary RADIUS enriched traffic log with 3GPP-specific attributes for enhanced RSSO solutions
Description [Customer Problem Statement (what are they trying to solve)]
Service Provider customers (Telcos - MNOs and fixed-line Operators) use RSSO to apply subscriber-based security services. This is well documented and understood. However, the limited RSSO attributes available as well as the nature of how RSSO identifies and “logs” users means that very limited subscriber information is available. In order to better apply the services offered, and offer more value, our Telco customers would like a greater level of subscriber information in the FortiGate logs to provide for richer analytics and reporting.

This NFR seeks to request an extra field in the FortiOS traffic log whereby additional RSSO data can be captured and logged, allowing for richer upstream analytics and reporting functions. There are a number of 3GPP specific attributes which are provided via RADIUS accounting messages, and it is these attributes along with the others provided for RSSO purposes which should be logged, namely:

* Calling-Station-ID(31) -> this will be used to supply the MSISDN and is generally used for for the “user” match in RSSO
* 3GPP-IMSI(1) -> will provide the IMSI of the subscriber
* Called-Station-ID(30) -> This is often used to convey the APN to which the subscriber is attached
* 3GPP-Session-S-NSSAI(125) - > While more specific to 5G networks, this will provide the network slice ID that the subscriber has assigned
* 3GPP-IMEISV(20) -> provides the device identity, most relevant for devices that don’t have users e.g. IoT devices
* 3GPP-RAT-Type(21) -> indicates the Radio Access Technology type that is in use e.g. UMTS, LTE, 5GNR, NB-IOT etc
* 3GPP-User-Location-Info(22) -> provides UE location information such as CAI and/or SAI
* Class(25) -> Used to convey various open attributes for the RSSO Group match, though generally this is used to allow for group/profile match in RSSO

The desire here is for RADIUS accounting messages to be recieved by FortiOS and for it to extract and apply the defined sso attribute as normal. But FortiOS should also capture and insert the attribute values of the attributes listed above, into the traffic log. It is expected that a new additional log field is added to the FortiOS traffic log, as no existing field seems viable for reuse for this need (though if Engineering have alternative options/suggestions for existing traffic log field reuse that can deliver the information needed, this can be an option to explore).

Ultimately the goal here is to receive the additional RADIUS attributes in FortiOS, which will log the values of all attributes listed above, in the traffic log. The traffic log is sent to log destinations such as FAZ, or generic SYSLOG destinations such as FSIEM or 3rd party SYSLOG collector/s. The logging/analytics platform (FAZ, FSIEM, or 3rd party) can then use this additional information to enrich any analytics activity such as dashboards and reporting outputs.

[Implementation Suggestions]

This NFR calls for an additional log field in the traffic log. This new log field should have the following characteristics:

* Log Field name = rsso-3gppinfo [this is an example name, engineering is free to define a more suitable name as needed]
* Description = 3GPP subscriber information from RADIUS [this is an example description, engineering is free to define a more suitable description as needed]
* Data Type = String [this seem most appropriate considering the various types of attributes this needs to display]
* Length = 512 [this will contain multiple entries and should capture number of fields so a large length is needed]

As this single log field needs to capture lots of information from all of the attributes recieved from RADIUS, it is expected that this log field will concatenate the different RADIUS attribute values using a delimiter, for example a comma. However, it is assumed that engineering will decide the most appropriate method here.

Furthermore, this enriched logging function should be configureable. A suggestion for configuration is mentioned blow. It should be noted that such a setting should honour VDOM configuration, therefore allowing for this “enriched RSSO logging” to be applied, or not, per VDOM.

Suggested configuration, default behaviour should be disabled:

config system log setting
  set extended-3gpp-radius-log [enable | *disable*]
end

[ Reply ] 		Relevant updates from TOP3 1229978:

The customer will be using RSSO to be able to apply UserFW and UTM profiles to subscriber groups. While this does not affect this logging requirement as such, it does mean that the Framed IP/Framed-IPv6 will get recorded in the traffic log anyway as "srcip" so we can remove these. But we need to support at least two potential Class(25) attributes as RSSO can support up to two. So the number of attributes we are capturing is reduced to 9 as follows

--Calling-Station-ID(31), 3GPP-IMSI(1), Called-Station-ID(30), 3GPP-Session-S-NSSAI(125), 3GPP-IMEISV(20), 3GPP-RAT-Type(21), 3GPP-User-Location-Info(22), Class(25)--, Class(25)--

Where an attribute is not provided, this should simply be skipped. It may be the case that only one Class(25) is used/provided so the RADIUS Acct message only contains one. Or it may contain two.

Also, attributes that have values which are not easy to read/understand (such as octets type - e.g. User-Location-Info, S-NSSAI and Class) should provide hexdump of the value. This ensures better representation of the variations in potential values that these different attributes may have. You will see my example below which indicates this.
